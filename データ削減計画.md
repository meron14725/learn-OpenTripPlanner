# データ削減計画：東京・埼玉・神奈川・千葉に絞り込む

**目的**: OTP2 グラフビルドのメモリ使用量・ビルド時間を削減するため、対象エリアを 4都県（東京・埼玉・神奈川・千葉）に限定する。

---

## 方針

### なぜ削るか

| 現状の問題 | 影響 |
|-----------|------|
| OSM: 関東全域（約700MB） | グラフビルドに大量メモリ・時間が必要 |
| GTFS: JR東日本が茨城・栃木・群馬まで収録 | 使わない駅・経路がグラフに含まれる |

### 対象エリアの定義（バウンディングボックス）

```
west:  138.90（神奈川西部）
east:  140.90（千葉東部）
south: 35.00（神奈川南部）
north: 36.30（埼玉北部）
```

> ⚠️ 埼玉北部（熊谷・本庄付近）と群馬の境界は緯度で綺麗に分離できない。
> より正確に切るなら **県境ポリゴン**（後述）を使う。

## 方法

### Step 1: OSM PBF の差し替え

#### 方法A（推奨）: Geofabrik 都県別 PBF を結合

```bash
# 各都県を個別にダウンロード
curl -L https://download.geofabrik.de/asia/japan/tokyo.osm.pbf -o data/osm/tokyo.osm.pbf
curl -L https://download.geofabrik.de/asia/japan/kanagawa.osm.pbf -o data/osm/kanagawa.osm.pbf
curl -L https://download.geofabrik.de/asia/japan/saitama.osm.pbf -o data/osm/saitama.osm.pbf
curl -L https://download.geofabrik.de/asia/japan/chiba.osm.pbf -o data/osm/chiba.osm.pbf

# osmium-tool で結合
osmium merge \
  data/osm/tokyo.osm.pbf \
  data/osm/kanagawa.osm.pbf \
  data/osm/saitama.osm.pbf \
  data/osm/chiba.osm.pbf \
  -o data/4pref.osm.pbf
```

> `osmium merge` はインストールが必要: `brew install osmium-tool` / `apt install osmium-tool`

#### 方法B（簡易）: 既存 PBF をバウンディングボックスでクリップ

```bash
osmium extract \
  --bbox 138.90,35.00,140.90,36.30 \
  data/kanto-latest.osm.pbf \
  -o data/4pref.osm.pbf
```

差し替え後、`build-config.json` の参照先が `4pref.osm.pbf` を参照するよう確認する（OTP2 は `data/` 内の `*.osm.pbf` を自動検出するため、旧 PBF を削除して新 PBF を配置すれば OK）。

---

### Step 2: GTFS の停留所フィルタリング

Python スクリプトで、各 GTFS zip の `stops.txt` を bbox 内に絞り込み、連鎖的に `stop_times.txt` → `trips.txt` → `routes.txt` を削除する。

#### フィルタリングロジック（概要）

```
bbox = (138.90, 35.00, 140.90, 36.30)

stops.txt → bbox 内の stop_id のみ残す
stop_times.txt → 残った stop_id の行のみ残す
trips.txt → 残った stop_times の trip_id のみ残す
routes.txt → 残った trips の route_id のみ残す
```

> **注意**: あるトリップの「一部の停留所」が bbox 外に出る場合、そのトリップごと削除される。
> 例: 宇都宮線の上野→宇都宮は「大宮以北（埼玉まで）」のトリップのみ残り、さらに北へ行くトリップは削除される。

#### スクリプト配置先

```
scripts/
└── filter_gtfs.py   # 新規作成
```

使い方:

```bash
python scripts/filter_gtfs.py \
  --input  data/GTFS-data/JR-East-Train.gtfs.zip \
  --output data/GTFS-filtered/JR-East-Train.gtfs.zip \
  --bbox   138.90,35.00,140.90,36.30
```

全ファイルに一括適用:

```bash
for f in data/GTFS-data/*.gtfs.zip; do
  python scripts/filter_gtfs.py \
    --input "$f" \
    --output "data/GTFS-filtered/$(basename $f)" \
    --bbox 138.90,35.00,140.90,36.30
done
```

---

### Step 3: OTP2 再ビルド

```bash
# data/ に 4pref.osm.pbf と GTFS-filtered/*.gtfs.zip を配置
# （GTFS-filtered/ をそのまま使うか、GTFS-data/ と入れ替える）

java -Xmx4G -jar otp.jar --build --save data/
```

削減後は `-Xmx4G` → `-Xmx3G` 程度に下げられる可能性がある。

## 懸念事項・注意点

### JR東日本のフィルタで注意が必要な路線

| 路線 | 状況 | 対処方針 |
|------|------|---------|
| 宇都宮線（上野〜宇都宮） | 上野〜小山まで埼玉、小山以北が栃木 | 小山以北のトリップを削除。上野〜大宮区間は湘南新宿ラインと重複するため許容範囲 |
| 高崎線（上野〜高崎） | 熊谷まで埼玉、深谷以北〜群馬 | 深谷以北を削除 |
| 常磐線（上野〜仙台） | 我孫子・取手まで千葉、荒川沖以北が茨城 | 取手以北を削除 |
| つくばエクスプレス | 秋葉原〜つくば、守谷が茨城 | 守谷以北（つくば）のトリップを削除 |

### フィルタ後に「孤立トリップ」が生じる可能性

bbox の境界をまたぐトリップは丸ごと削除されるため、**境界付近の折り返し列車**（例：大宮止まりの宇都宮線）は残るが、**終点が bbox 外のトリップ**は消える。
→ 実用上は「東京方面への通勤」しか検索しないため影響は少ない。

### 方法B（bbox clip）の制限

`osmium extract --bbox` は「厳密な県境」ではなく矩形でカットするため、埼玉北部（秩父方面）が入り、群馬の一部が入る可能性がある。
OSM 側はこれで許容範囲。問題になるのは GTFS 側なので、GTFS は別途フィルタする。
