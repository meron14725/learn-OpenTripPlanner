## OSM PBF 削減

### 実施方法

```bash
osmium extract \
  --bbox 138.90,35.00,140.90,36.30 \
  data/osm/kanto-latest.osm.pbf \
  -o data/osm/4pref.osm.pbf
```

処理時間: **16秒**

### 計測結果

| 指標 | kanto-latest | 4pref | 削減数 | 削減率 |
|------|-------------|-------|--------|--------|
| ファイルサイズ | 437.2 MB | 365.5 MB | -71.7 MB | **-16.4%** |
| ノード数 | 57,055,845 | 48,657,771 | -8,398,074 | -14.7% |
| ウェイ数 | 9,172,414 | 8,163,473 | -1,008,941 | -11.0% |
| リレーション数 | 67,024 | 60,849 | -6,175 | -9.2% |

### 考察

削減率が 16% にとどまった理由:
- 関東 OSM は東京・神奈川に極端にデータが集中しており、茨城・栃木・群馬は道路・建物密度が相対的に低い
- 切り取る bbox の北限を 36.30°N に設定したため、埼玉北部（熊谷・本庄）はカバーできているが、群馬南部の一部が入る可能性がある

## GTFS フィルタの代替アプローチ（将来検討）

「bbox 外の停留所を含むトリップを全削除」ではなく、**bbox 外の停留所のみ削除して、残りの停留所でトリップを継続させる**手法（stop trimming）があるが:

- OTP2 の GTFS 仕様上、trip の shape_dist_traveled 整合が崩れる可能性がある
- 実装コストが高い
- 今回のユースケース（都内ルーティング）では大きなメリットがない

→ **GTFS フィルタは今回は適用しない。GTFS は現状 22 事業者を全てそのまま使う。**

## 最終方針と推奨構成

| 項目 | 方針 |
|------|------|
| OSM | `4pref.osm.pbf` (365.5MB) を使用する |
| GTFS | 22事業者全て現状維持（フィルタなし） |
| graph build | OSM のみ削減した状態で `--build --save` |

OSM の物理ファイルサイズは 16% 削減できるが、OTP2 のグラフビルド時のメモリ使用量への影響は **OSM ノード数削減率（14.7%）に比例**すると想定される。

## グラフビルド実測結果

OTP2 v2.8.1 を使用して実際にビルドを実施し、メモリ・時間を計測した。

### 計測環境

- OTP2: v2.8.1
- Java: OpenJDK 21.0.10
- RAM: 22 GB (MemAvailable)
- 計測スクリプト: `scripts/measure_build.sh`（RSS を 5 秒おきにポーリング）

### 実測値（-Xmx8G）

| 指標 | kanto baseline | 4pref (削減後) | 削減率 |
|------|---------------|----------------|--------|
| ビルド時間 | 12分18秒 (738秒) | 10分8秒 (608秒) | **-17.7%** |
| ピーク RSS | 8,817 MB (8.6 GB) | 8,114 MB (7.9 GB) | **-8.0%** |
| graph.obj サイズ | 651 MB | 540 MB | **-17.1%** |
| グラフ頂点数 | 3,813,317 | 3,170,185 | -16.9% |
| グラフ辺数 | 10,386,981 | 8,629,005 | -16.9% |
| 街路グラフノード数 | 2,408,790 | 1,945,524 | **-19.2%** |
| 停留所数 | 2,060 | 2,060 | 0%（GTFS フィルタなし） |
| トリップ数 | 84,324 | 84,324 | 0% |

### 最小 -Xmx の探索

4pref データセットで -Xmx を段階的に下げてビルドが成功するか確認した。

| -Xmx | 結果 | ビルド時間 | graph.obj |
|------|------|-----------|-----------|
| 8G | ✅ SUCCESS | 10分8秒 | 540 MB |
| 7G | ✅ SUCCESS | 10分5秒 | 540 MB |

→ **4pref データセットは -Xmx7G でビルド可能**（ビルド時間の差はほぼなし）

**推奨 -Xmx**: 安全マージンを考慮し **8G** を推奨（ピーク RSS ~7.9GB）

### 考察

- OSM 削減（-16.4%）がグラフビルド時間に直結し **-17.7% の短縮**を実現
- ピーク RSS の削減率（-8.0%）は OSM ノード数の削減率（-14.7%）より低い
  → GTFS の 22 事業者（停留所 2,060・トリップ 84,324）のロードは OSM と無関係のため
- ピーク RSS は JVM ヒープ（-Xmx で指定した上限）ではなく OS レベルの実メモリ消費
  → baseline (-Xmx8G) でピーク RSS = 8.6GB なので、JVM の Off-Heap 分が加わっている

## サーバーモード最小メモリ計測（--load）

4pref の `graph.obj`（540MB）を使って OTP2 をサーバーモードで起動し、最小 -Xmx を探索した。

### テスト条件

- コマンド: `java -Xmx<X> -jar otp.jar --load data/build-4pref/`
- 確認内容:
  - 起動成功か OOM か
  - 起動時間（秒）
  - アイドル RSS（起動完了直後）
  - クエリ後 RSS（GraphQL リクエスト後）
  - GraphQL 応答: 高円寺→渋谷の `planConnection` クエリで3経路返却されるか

### 計測結果

| -Xmx | 起動 | 起動時間 | アイドル RSS | クエリ後 RSS | GraphQL 応答 |
|------|------|---------|------------|------------|------------|
| 2G | ✅ SUCCESS | 56秒 | 2,823 MB | 2,902 MB | ✅ 3経路返却 |
| 1,800m | ✅ SUCCESS | 58秒 | 2,481 MB | 2,557 MB | ✅ 3経路返却 |
| 1,700m | ✅ SUCCESS | 62秒 | 2,343 MB | 2,429 MB | ✅ 3経路返却 |
| 1,600m | ❌ OOM | - | - | - | - |
| 1,500m | ❌ OOM | - | - | - | - |

- `OOM` = `java.lang.OutOfMemoryError: Java heap space` (graph.obj のデシリアライズ中)

### 最小 -Xmx

**-Xmx1700m（1.7GB）** でサーバー起動・GraphQL 応答ともに成功。
**推奨: -Xmx2G** （安全マージンを確保、クエリ処理で ~2.9GB RSS を消費）

### 考察

- graph.obj は 540MB だが、デシリアライズ後のメモリは大幅に膨張する（Java ヒープ内でオブジェクトグラフとして展開されるため）
- RSS = 2,343〜2,902 MB は JVM ヒープ（-Xmx の上限） + JVM の Off-Heap（Native memory, Metaspace, Stack など）
- **ビルド時（7.9GB）の約 1/3** でサーバー稼働可能であることが確認できた

高円寺（35.7037°N, 139.6494°E）→ 渋谷（35.6581°N, 139.7014°E）
